## 1. 每次开发会话的使用方式

每次实现会话开始前，Agent 必须按顺序完成：

1. 阅读 `docs/v0.2.11/README.md`。
2. 阅读本执行计划。
3. 如果已经存在目标主题目录，完整阅读该主题的 final design 和 execution plan。
4. 检查第 4 节版本级台账，确认当前可认领事项。
5. 检查 `git status --short`，保留用户现有改动，不覆盖无关文件。
6. 在编辑前声明本次认领范围、预期改动文件、验证命令和明确不涉及的范围。

每次实现会话结束前必须：

1. 运行相关验证，或准确记录无法运行的原因。
2. 更新第 4 节版本级台账，以及对应主题 execution plan 的工作包台账。
3. 在对应主题的 implementation records 目录新增或更新实施记录。
4. 只有代码、测试、文档和验证均闭环时，才标记为 `已完成`。
5. 如果实现证明设计假设不成立，先更新 final design，或创建 `feat/` / `fix/` 文档，不能静默偏离。
6. 回答用户前结束本次会话启动的全部 Vite、Electron 或其他前端服务进程。

不得把聊天上下文作为唯一交接信息；下一位 Agent 应只依赖设计文档、执行计划、实施记录和代码即可继续。

## 2. 状态规则

工作包状态只允许使用：

- `未开始`
- `进行中`
- `已完成`
- `阻塞`
- `废弃`

状态解释：

- `未开始`：尚未产生实现或验证工作。
- `进行中`：已修改代码或开展验证，但尚未满足完整验收口径。
- `已完成`：实现、验证、实施记录和台账已经闭环。
- `阻塞`：存在明确外部阻塞，当前会话无法继续推进。
- `废弃`：经设计更新确认不再实施，必须记录替代方案或原因。

---

## 3. 版本推进原则

1. 先立项再实现：每个中大型需求都应先有 final design 和 execution plan。
2. 先最小闭环再扩展：优先完成可验证的端到端路径，再补体验、兼容和边界。
3. 文档和代码同步：执行计划台账、实施记录和实际代码状态必须一致。
4. 不混改无关内容：跨主题修复应单独记录为 `fix/` 文档或独立工作包。
5. 验证命令要精确：记录实际执行的命令、结果和未覆盖风险。
6. 前端服务要清理：本版本开发遵守项目要求，回答结束前关闭本次启动的前端服务。

## 5. 新主题立项模板

收到具体需求后，在 `docs/v0.2.11/<feature-slug>/` 下创建：

```text
<feature-slug>_final_design.md
<feature-slug>_execution_plan.md
<feature-slug>_implementation_records/
feat/
fix/
```

final design 至少包含：

- 背景、目标、非目标。
- 当前代码状态和约束。
- 最终架构、模块职责和关键数据流。
- 前端交互、加载/空/错误状态。
- 后端或 Electron 主进程接口、持久化、权限和错误处理。
- 风险、边界、验证策略和发布说明。

execution plan 至少包含：

- 每次会话的使用方式。
- 状态规则。
- 里程碑。
- 进度台账。
- 可单会话闭环的工作包详情。
- 实施记录模板。
